# 🛡️ 03-Suricata-Alert-Analysis

---

# 🎯 Objective

The objective of this project is to learn how to investigate and analyze Suricata alerts using both **fast.log** and **eve.json**.

In previous modules:

- Suricata was successfully deployed
- Detection rules were investigated
- Custom detection rules were created and tested

This module focuses on the analyst perspective by answering questions such as:

- What happened?
- Which rule triggered?
- Who initiated the communication?
- What protocol was involved?
- Is the activity malicious or expected?
- How can the alert be validated?

By the end of this project, Suricata alerts can be investigated using a structured SOC analyst workflow.

---

# 🧠 Understanding Suricata Logs

Suricata generates multiple log files containing different types of information.

The most important logs for alert investigation are:

| Log File | Purpose |
|-----------|-----------|
| fast.log | Human-readable alert log |
| eve.json | Structured JSON event log |
| stats.log | Engine statistics and performance metrics |

Each log serves a different purpose and provides a different level of visibility.

---

# 📋 Prerequisites

Before starting this module:

- Suricata must be installed and running
- Emerging Threats rules must be loaded
- Custom ICMP detection rule must be created
- Alert generation must already be validated
- Access to Suricata log files must be available

---

# 🔍 Exploring fast.log

The fast.log file provides a simplified view of Suricata alerts.

To review recent alerts:

## Command

```bash
sudo tail -20 /var/log/suricata/fast.log
```

## Purpose

This file provides quick visibility into:

- Alert Name
- Signature ID (SID)
- Classification
- Severity
- Protocol
- Source IP
- Destination IP

It is designed for rapid human review and alert triage.

### 📸 Screenshot

![Fast Log Overview](images/01-fast-log-overview.png)

---

# 🔍 Exploring eve.json

The eve.json file provides structured event data generated by Suricata.

To review recent events:

## Command

```bash
sudo tail -5 /var/log/suricata/eve.json
```

## Purpose

Unlike fast.log, eve.json provides:

- Flow Information
- Packet Counts
- Byte Counts
- Protocol Metadata
- Alert Metadata
- Traffic Direction
- Application Protocol Identification

This format is commonly ingested into SIEM platforms such as:

- Microsoft Sentinel
- Splunk
- Elastic
- QRadar

### 📸 Screenshot

![EVE JSON Overview](images/02-eve-json-overview.png)

# Why Analysts Need Both Logs

Both logs serve different purposes.

## fast.log

Best for:

- Quick alert review
- Manual investigation
- Rapid triage

---

## eve.json

Best for:

- Deep investigations
- SIEM ingestion
- Correlation
- Threat hunting

A SOC analyst often starts with fast.log and then pivots into eve.json for additional context.

# 🔎 Custom ICMP Alert Investigation

The following alert was generated:

```text
CUSTOM RULE - ICMP Ping Detected
```

Example Alert:

```text
[1:1000001:1] CUSTOM RULE - ICMP Ping Detected
```

---

# Alert Breakdown

## Timestamp

The timestamp indicates when Suricata generated the alert.

Example:

```text
2026-06-07 16:44:52
```

---

## Signature Information

```text
[1:1000001:1]
```

Where:

| Field | Value |
|---------|---------|
| GID | 1 |
| SID | 1000001 |
| REV | 1 |

This identifies the exact detection rule responsible for the alert.

---

## Alert Name

```text
CUSTOM RULE - ICMP Ping Detected
```

This value comes directly from the rule's message field:

```text
msg:"CUSTOM RULE - ICMP Ping Detected"
```

---

## Classification

```text
Classification: (null)
```

No classification was defined because the custom rule did not contain a:

```text
classtype:
```

field.

---

## Severity

```text
Priority: 3
```

This indicates a low-severity informational alert.

---

## Protocol

```text
IPv6-ICMP
```

The alert was generated from IPv6 ICMP traffic rather than traditional IPv4 ICMP traffic.

---

## Source Address

```text
fe80::
```

A link-local IPv6 address assigned to the Linux host.

---

## Destination Address

```text
ff02::2
```

An IPv6 multicast address used to communicate with all IPv6 routers on the local network.

---

# Analyst Assessment

The custom ICMP alert was generated by expected IPv6 network activity.

No evidence of malicious behavior was observed.

The alert successfully validated that:

```text
Traffic
    ↓
Rule Match
    ↓
Alert Generation
    ↓
Logging
```

was functioning correctly.

# Investigating the Same Alert in eve.json

To view the structured version of the alert:

## Command

```bash
grep "1000001" /var/log/suricata/eve.json | tail -1
```

---

# Additional Context Available

Unlike fast.log, eve.json provides:

- Flow ID
- Packet Counts
- Byte Counts
- Direction
- Interface Information
- ICMP Type
- ICMP Code

Example fields:

```json
"event_type":"alert"
"proto":"IPv6-ICMP"
"severity":3
```

---

# Key Findings

## Event Type

```json
"event_type":"alert"
```

Confirms that a detection was generated.

---

## Interface

```json
"in_iface":"eth0"
```

The traffic was observed on the monitored network interface.

---

## Flow ID

```json
"flow_id"
```

A unique identifier representing the network conversation.

This value can be used to correlate multiple events belonging to the same session.

---

## Protocol

```json
"proto":"IPv6-ICMP"
```

Confirms the protocol responsible for triggering the alert.

---

## Action

```json
"action":"allowed"
```

Because Suricata is operating in IDS mode, traffic is detected but not blocked.

---

# Analyst Conclusion

The structured JSON event confirms that the custom rule successfully detected IPv6 ICMP traffic.

The event represents expected operating system behavior and was intentionally generated to validate custom rule functionality.

# 🖥️ RDP Alert Analysis

The next alert investigated was generated by an Emerging Threats detection rule.

To locate the alert:

## Command

```bash
grep "2001330" /var/log/suricata/fast.log | tail -1
```

Example Alert:

```text
ET INFO RDP - Response To External Host
```

### 📸 Screenshot

![RDP Alert Fast Log](images/06-rdp-alert-fastlog.png)

---

# Alert Breakdown

## Signature Information

```text
[1:2001330:10]
```

| Field | Value |
|---------|---------|
| GID | 1 |
| SID | 2001330 |
| REV | 10 |

---

## Alert Name

```text
ET INFO RDP - Response To External Host
```

This rule detects Remote Desktop Protocol traffic responding to an external client.

---

## Classification

```text
Misc activity
```

This classification indicates noteworthy network activity rather than malicious behavior.

---

## Severity

```text
Priority: 3
```

Informational severity.

---

## Protocol

```text
TCP
```

RDP operates over TCP.

---

## Source

```text
10.0.0.5:3389
```

The Linux VM running XRDP.

Port:

```text
3389
```

is the standard Remote Desktop Protocol port.

---

## Destination

```text
103.150.139.130:55956
```

The external client connected to the Linux VM.

The high-numbered port represents a temporary client-side port.

---

# Initial Analyst Assessment

The alert indicates that the Linux VM responded to an external RDP client.

This activity aligns with legitimate administration of the system using XRDP.

No malicious behavior is immediately apparent.

---

# Investigating the Same Alert in eve.json

To view the structured event:

## Command

```bash
grep '"signature_id":2001330' /var/log/suricata/eve.json | tail -1
```

### 📸 Screenshot

![RDP Alert EVE JSON](images/07-rdp-alert-evejson.png)

---

# Additional Findings

## Application Protocol

```json
"app_proto":"rdp"
```

Suricata successfully identified the traffic as Remote Desktop Protocol.

---

## Session Direction

```json
"direction":"to_client"
```

Traffic was flowing from:

```text
Linux VM
    ↓
Remote Client
```

---

## Flow Duration

The flow start timestamp indicated that the session had already been active for several minutes before the alert was generated.

---

## Packet Statistics

```json
"pkts_toserver":18803
"pkts_toclient":36005
```

Total packets observed:

```text
54,808
```

---

## Data Transfer

```json
"bytes_toserver":1267293
"bytes_toclient":48083991
```

Approximately:

| Direction | Data |
|------------|---------|
| Client → Server | ~1.2 MB |
| Server → Client | ~48 MB |

This behavior is expected because RDP primarily sends screen updates from the server to the client.

---

## Flowbits

```json
"flowbits":["ms.rdp.established"]
```

This confirms that Suricata recognized a valid established RDP session.

---

## Retransmissions

```json
"tcp.retransmission.count":3
```

A small number of retransmissions is normal on real-world networks.

---

# Analyst Conclusion

The alert was generated from a legitimate XRDP administrative session.

Evidence supporting this conclusion:

- Valid RDP protocol identified
- Established session detected
- Large volume of screen-update traffic
- Expected source and destination hosts
- Informational severity

No malicious activity was observed.

```text
ET HUNTING ZIP file exfiltration over raw TCP
```

To locate the alert:

## Command

```bash
grep "2035478" /var/log/suricata/fast.log | tail -1
```

### 📸 Screenshot

![ZIP Alert Fast Log](images/08-zip-alert-fastlog.png)

---

# Alert Breakdown

## Signature Information

```text
[1:2035478:2]
```

| Field | Value |
|---------|---------|
| GID | 1 |
| SID | 2035478 |
| REV | 2 |

---

## Alert Name

```text
ET HUNTING ZIP file exfiltration over raw TCP
```

The alert name suggests that Suricata observed traffic containing characteristics of a ZIP archive.

---

## Classification

```text
Misc activity
```

Not classified as malware, exploitation, or confirmed data theft.

---

## Severity

```text
Priority: 3
```

Informational severity.

---

## Source

```text
10.0.0.5
```

Linux VM.

---

## Destination

```text
168.63.129.16
```

Azure platform infrastructure IP address.

---

# Initial Observation

Although the alert contains the term:

```text
Exfiltration
```

an analyst should never assume malicious activity based solely on the alert name.

Investigation and validation are required before escalation.

---

# Investigating the Same Alert in eve.json

To review the structured event:

## Command

```bash
grep '"signature_id":2035478' /var/log/suricata/eve.json | tail -1
```

### 📸 Screenshot

![ZIP Alert EVE JSON](images/09-zip-alert-evejson.png)

---

# Additional Findings

## Direction

```json
"direction":"to_server"
```

Traffic originated from the Linux VM and was sent toward the destination host.

---

## Packet Count

```json
"pkts_toserver":4
"pkts_toclient":1
```

Total packets observed:

```text
5
```

---

## Data Volume

```json
"bytes_toserver":2159
```

Approximately:

```text
2.1 KB
```

transferred.

---

## Destination Analysis

The destination IP:

```text
168.63.129.16
```

belongs to Microsoft Azure platform services.

Examples include:

- DHCP
- Platform communication
- Health monitoring
- Guest agent communication

This is significantly different from sending data to an unknown Internet host.

---

## Rule Logic

The signature searches for ZIP archive headers.

Specifically:

```text
PK 03 04
```

which represents the beginning of a ZIP file.

The rule therefore indicates:

```text
Traffic matched a ZIP archive pattern
```

and does not automatically confirm successful data exfiltration.

---

# Analyst Conclusion

The investigation revealed:

- Low data volume (~2 KB)
- Azure infrastructure destination
- Informational severity
- ZIP header pattern match

Current evidence does not support malicious data exfiltration.

The alert is most likely a false positive or expected Azure platform communication that matched the detection signature.
